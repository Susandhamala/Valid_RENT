Algorithm Breakdown
RSA-2048 Key Generation
Asymmetric · One pair per user
Why

RSA is asymmetric — the public key is freely shared, the private key stays secret. 2048 bits gives 112-bit security (NIST minimum). Exponent 65537 is prime with only two set bits, making modular exponentiation fast while blocking low-exponent attacks.

What the code does

1. OS CSPRNG picks two large primes → compute n=p×q and private exponent d. 2. Private key → PKCS#8 PEM (no passphrase in demo). 3. Public key → SubjectPublicKeyInfo PEM. Both stored in users table; public key passed straight to issue_certificate().

def generate_rsa_keypair():
    # OS CSPRNG → RSA-2048, e=65537
    private_key = rsa.generate_private_key(
        public_exponent=65537, key_size=2048, backend=default_backend()
    )
    # PKCS#8 PEM (no passphrase in demo)
    private_pem = private_key.private_bytes(
        Encoding.PEM, PrivateFormat.PKCS8, NoEncryption()
    ).decode('utf-8')
    # SubjectPublicKeyInfo PEM
    public_pem  = private_key.public_key().public_bytes(
        Encoding.PEM, PublicFormat.SubjectPublicKeyInfo
    ).decode('utf-8')
    return private_pem, public_pem
RSA-PSS + SHA-256 Signature
Digital signing · Non-repudiation
Why PSS, not PKCS#1 v1.5?

PSS is provably secure under the RSA assumption. A random 222-byte salt means signing the same document twice gives different bytes — preventing replay and existential forgery. PKCS#1 v1.5 is vulnerable to Bleichenbacher-style padding attacks.

What the code does

Sign: Hex hash decoded to 32 bytes → private_key.sign(PSS+MGF1+SHA256) — applies randomised masking + RSA private exponentiation. Result Base64-encoded for DB. Verify: public_key.verify() raises InvalidSignature on any mismatch; only returns True when math checks out.

def rsa_sign(private_key_pem, data_hex):
    data = bytes.fromhex(data_hex)      # 32 bytes from SHA-256
    sig  = load_private_key(private_key_pem).sign(
        data,
        padding.PSS(mgf=padding.MGF1(hashes.SHA256()),
                     salt_length=padding.PSS.MAX_LENGTH),  # 222B salt
        hashes.SHA256()
    )
    return base64.b64encode(sig).decode()

def rsa_verify(public_key_pem, data_hex, sig_b64):
    try:
        load_public_key(public_key_pem).verify(
            base64.b64decode(sig_b64), bytes.fromhex(data_hex),
            padding.PSS(mgf=padding.MGF1(hashes.SHA256()),
                         salt_length=padding.PSS.MAX_LENGTH),
            hashes.SHA256()
        )
        return True
    except: return False  # tampered / wrong key
AES-256-GCM Encryption
Symmetric AEAD · Files + Chat
Why GCM?

GCM is an AEAD cipher — confidentiality plus a 128-bit integrity tag in one pass. A fresh os.urandom(12) nonce per call keeps nonce-collision probability at 2⁻⁶⁴. Reusing a nonce with the same key in GCM is catastrophic (leaks the auth key).

What the code does

File: Generate 256-bit key + 12-byte nonce → AESGCM.encrypt() returns ciphertext with 16-byte tag appended. Key + nonce returned to caller for RSA-OAEP wrapping. Chat: Thread key wrapped in master-key GCM (nonce prepended). Each message encrypted independently with its own nonce before DB insert.

def encrypt_file_aes256gcm(file_path, output_path):
    aes_key = AESGCM.generate_key(bit_length=256)  # 32 random bytes
    nonce   = os.urandom(12)                       # fresh every call
    ct = AESGCM(aes_key).encrypt(nonce,
             open(file_path,'rb').read(), None)
    open(output_path,'wb').write(ct)
    return b64(aes_key), b64(nonce)   # caller wraps key with RSA-OAEP

# chat — per-message
def encrypt_message(thread_key_b64, plaintext):
    key   = _decrypt_thread_key(thread_key_b64)
    nonce = os.urandom(12)
    ct    = AESGCM(key).encrypt(nonce, plaintext, None)
    return b64(ct), b64(nonce), sha256(plaintext)
SHA-256 Hash Function
Cryptographic hash · 6 call sites
Why SHA-256?

One-way with 128-bit collision resistance — one changed byte flips ~half the output (avalanche). Used both as a tamper detector and a KDF (stretches any string to exactly 32 bytes for AES-256). Also the digest inside RSA-PSS and MGF1.

6 call sites

1. Document hash — all fields hashed; RSA signs this, not the raw text. 2. Master KDF — sha256(SECRET_KEY).digest() → 32-byte AES master key. 3. File hash — 8 KB chunks, no OOM. 4. Photo hash — JPEG bytes hashed at capture. 5. Message hash — hashed before encryption for integrity check on decrypt. 6. PDF hash — printed inside the PDF itself.

# chunked file hash — any file size, no OOM
def sha256_hash_file(file_path):
    h = hashlib.sha256()
    with open(file_path, 'rb') as f:
        for chunk in iter(lambda: f.read(8192), b''):
            h.update(chunk)
    return h.hexdigest()          # 64-char hex

# master AES key — stretch SECRET_KEY to 32 bytes
def _master_key():
    return hashlib.sha256(SECRET_KEY.encode()).digest()
RSA-OAEP Key Wrapping
Hybrid encryption · AES key transport
Why hybrid?

RSA-2048 can only encrypt up to 214 bytes — it cannot encrypt a file directly. AES is fast but needs a shared secret. Solution: AES-GCM encrypts the file; RSA-OAEP encrypts only the 32-byte AES key. OAEP uses a random seed + MGF1-SHA256 masking, blocking Bleichenbacher's 1998 CCA attack against PKCS#1 v1.5.

What the code does

Wrap: public_key.encrypt(aes_key, OAEP(MGF1+SHA256)) — OAEP generates a random seed, masks it with MGF1, then RSA public exponentiation (data^e mod n). Output: 256-byte ciphertext, Base64. Unwrap: Only the matching private key can run the inverse and recover the AES key.

def encrypt_rsa_oaep(public_key_pem, plaintext_bytes):
    ct = load_public_key(public_key_pem).encrypt(
        plaintext_bytes,
        padding.OAEP(mgf=padding.MGF1(hashes.SHA256()),
                       algorithm=hashes.SHA256(), label=None)
    )
    return base64.b64encode(ct).decode()

def decrypt_rsa_oaep(private_key_pem, ciphertext_b64):
    return load_private_key(private_key_pem).decrypt(
        base64.b64decode(ciphertext_b64),
        padding.OAEP(mgf=padding.MGF1(hashes.SHA256()),
                       algorithm=hashes.SHA256(), label=None)
    )  # returns 32-byte AES key
X.509 Certificates & PKI
Identity binding · CA trust chain
Why X.509?

A raw RSA public key proves nothing about who owns it. An X.509 certificate binds a public key to name + email via a CA's signature, asserting identity for 1 year. On every QR scan both cert expiry and CA signature are re-verified from scratch — no trust caching.

What the code does

CA — _get_or_create_ca(): RSA-2048 self-signed cert, BasicConstraints(ca=True), 10-year validity. User cert — sets subject CN+email, embeds public key, ca=False, signed by CA key (SHA256). Verify — check not_valid_after < now, then ca_pubkey.verify(cert.signature, tbs_bytes).

def issue_certificate(user_public_key_pem, full_name, email):
    cert = (
        x509.CertificateBuilder()
        .subject_name(x509.Name([
            x509.NameAttribute(NameOID.COMMON_NAME,   full_name),
            x509.NameAttribute(NameOID.EMAIL_ADDRESS, email),
        ]))
        .issuer_name(ca_cert.subject)    # Demo CA signs it
        .public_key(public_key)          # user's RSA-2048 key
        .not_valid_after(now + timedelta(days=365))
        .add_extension(x509.BasicConstraints(ca=False,...), critical=True)
        .sign(ca_key, hashes.SHA256(), default_backend())
    )
